# 윌로그 영업팀 업무 현황 트래커 스킬

영업 담당자들이 각자 맡은 영업 프로젝트(고객사/사업 건)와 내부 업무가 어떻게 진행되고 있는지를
Gmail·Slack·Calendar·Pipedrive·Confluence 데이터를 기반으로 분석하여,
영업 트랙과 내부 업무 트랙을 분리한 담당자별 현황 리포트를 생성합니다.

---

## 🧭 본 스킬의 최상위 2대 원칙 (The Two Laws)

아래 두 원칙은 본 스킬의 모든 단계·모든 절차보다 **항상 우선**합니다. 이 두 원칙과 세부 지침이 충돌하면 항상 이 두 원칙이 이깁니다.

### 제1원칙: 완전 수집 (Full Collection)
**데이터는 모두 수집한다.** 지정 기간 내 Gmail·Slack·Calendar·Pipedrive·Confluence 다섯 채널의 해당 데이터를 빠짐없이 수집한다. 토큰·크기 한계는 분할·재시도·필터링으로 해결할 뿐, "일부만 수집"은 허용되지 않는다.

### 제2원칙: 추론 금지 (Facts Only)
**확인된 "사실"만 취급한다.** 수집된 원본 데이터에서 직접 확인된 사실만 리포트에 기재한다. 맥락·경험·상식에 기반한 추론·추정·창작은 일체 금지한다. 확인되지 않은 정보는 "확인 불가"로 표기하거나 해당 항목 자체를 생략한다.

> ⚠️ **AI 영업 전략 컨설팅(섹션 1-4) 및 내부 업무 코멘트(섹션 2-3)는 제2원칙의 예외 영역**입니다.
> 해당 섹션에서는 수집된 사실을 바탕으로 전략적 판단과 행동 제안을 능동적으로 작성합니다.
> 단, 제안의 근거가 되는 사실은 반드시 수집된 원본 데이터에서 확인된 것이어야 합니다.

## 1단계: 분석 범위 확인

스킬 실행 직후 사용자에게 다음 정보를 한 번에 모아서 질문합니다.

**필수 수집 정보:**

1. **분석 시작 시점** — 언제부터의 데이터를 분석할까요? 기본값은 **1주일 전(7일 전)**입니다. 별도로 지정하지 않으면 현재 시각 기준 7일 전부터 분석합니다. (예: "3일 전", "2026년 3월 10일", "지난 월요일 오전 9시")
2. **분석 대상 인원** (선택) — 위 6명 전체를 분석할까요, 아니면 특정 인원만 볼까요? 이름 또는 이메일로 알려주세요. 답하지 않으면 전원 분석합니다.
3. **집중 분석 대상 프로젝트 / 고객사** (선택) — 특별히 깊게 봐야 할 고객사나 프로젝트가 있나요? (예: "삼성SDS VN", "쿠팡FS", "이안로지스") 답하지 않으면 기본값은 **전체 프로젝트 분석**입니다.

> 분석 종료 시점은 스킬이 실행되는 현재 시각입니다. 별도로 묻지 않아도 됩니다.

## 2단계: 데이터 수집

사용자가 응답하면 Gmail·Slack·Calendar·Pipedrive·Confluence에서 데이터를 병렬로 수집합니다.

> ### 🚨 완전 수집 원칙 (제1원칙의 적용)
>
> **지정 기간 내 다섯 채널의 모든 해당 데이터를 빠짐없이 수집해야 합니다. 일부만 수집한 상태로 분석·리포트 단계로 넘어가는 것을 금지합니다.**
>
> #### 필수 준수 사항
>
> 1. **응답 크기 한계로 인한 조회 실패는 "수집 실패"가 아닙니다.** 반드시 분할·재시도로 완전 수집을 완료해야 합니다.
>    - 한 번의 API 호출 응답이 토큰·크기 한계를 초과하면 **페이지네이션(limit/start/cursor), 기간 분할(일·주 단위), 담당자·조직·파이프라인 등 축 단위 필터링**으로 쿼리를 쪼개 재호출합니다.
>    - 원본 응답을 파일로만 저장하고 넘어가지 않습니다. 저장한 파일은 반드시 끝까지 읽어 구조화된 데이터로 변환합니다.
>    - 한 번의 시도가 실패하면 전략을 바꿔(쿼리 축 변경, 조건 세분화, 범위 축소) 반드시 재시도합니다. "수집 불가"라는 결론은 모든 합리적 분할 전략을 시도한 뒤에만 내릴 수 있습니다.
>
> 2. **채널별 완전 수집 체크리스트 — 수집 종료 전에 모두 만족해야 함:**
>
>    **Gmail**
>    - [ ] 분석 대상 6인 전원에 대해 `from:<email>` 쿼리로 기간 내 발신 메일 전량 수집
>    - [ ] 분석 대상 6인 전원에 대해 `to:<email> OR cc:<email>` 쿼리로 기간 내 수신 메일 전량 수집
>    - [ ] 프로젝트·고객사·이슈 키워드 기반 보조 검색 수행
>    - [ ] 각 스레드의 본문·첨부·참여자 메타데이터 확보
>
>    **Slack**
>    - [ ] 분석 대상 6인 전원에 대해 `from:@<user>` 쿼리로 기간 내 메시지 전량 수집
>    - [ ] 6인 전원에 대해 멘션(`@<user>`) 메시지 수집
>    - [ ] DM·그룹DM 포함, 공개·비공개 채널 모두 포함
>    - [ ] 이슈·고객사·내부 프로젝트 키워드 기반 보조 검색 수행
>
>    **Google Calendar**
>    - [ ] 6인 각각의 calendarId로 기간 내 전체 이벤트 조회
>    - [ ] 각 이벤트의 organizer·attendees·location·description 메타데이터 확보
>    - [ ] ⚠️ "All-Hands" / "올핸즈" 이벤트는 수집 후 분석에서 제외 (수집 자체는 수행)
>
>    **Pipedrive**
>    - [ ] `get_users`로 6인의 user_id 매핑 테이블 확보 (담당자 매핑의 Ground Truth)
>    - [ ] `get_pipelines`로 파이프라인 메타 확보
>    - [ ] **owner별 `get_deals(owner_id)` 쿼리를 6인 전원에 대해 실행** — 이것이 담당자-딜 매핑의 1차 근거
>    - [ ] 딜 수가 많아 응답이 잘리면 페이지네이션(limit/start)으로 끝까지 조회
>    - [ ] 각 딜의 `get_deal_notes`, `get_activities`까지 모두 수집
>    - [ ] 특정 고객사·이슈가 식별되면 `search_deals`, `get_deal(id)`로 보강 조회
>
>    **Confluence**
>    - [ ] `PULSE TEAM` 스페이스의 기간 내 생성·수정된 페이지 전량 수집 (`searchConfluenceUsingCql` 사용)
>    - [ ] CQL 예시: `space = "PULSE TEAM" AND lastModified >= "YYYY-MM-DD" ORDER BY lastModified DESC`
>    - [ ] 각 페이지의 제목·본문·작성자·수정자·수정일시 메타데이터 확보
>    - [ ] 페이지 수가 많아 응답이 잘리면 `cursor` 페이지네이션으로 끝까지 조회
>
> 3. **"데이터가 너무 많아서 일부만"이라는 판단 금지.** 기술적 제약이 발생하면 사용자에게 누락을 알리고 넘어가는 것이 아니라, **쿼리 분할·재시도**로 해결합니다. 모든 분할 전략을 시도한 뒤에도 수집되지 않은 항목이 남으면, 해당 항목명·사유·시도한 전략을 4단계 정합성 검증에 반드시 기록합니다.
>
> 4. **수집 종료 선언**은 위 체크리스트를 모두 "수집 완료"로 표시한 뒤에만 할 수 있습니다. 이를 만족하지 않은 상태에서 3단계(분석)로 진입하는 것을 금지합니다.
>
> 5. **분할 수집 실패 시 즉시 사용자 보고** — 데이터 양이 많아 분할 수집을 시도했으나 분할 수집마저 실패한 경우, 수집을 중단하고 **즉시 사용자에게 아래 형식으로 보고**합니다. 분석 단계로 절대 진입하지 않습니다.
>
>    **보고 형식:**
>    ```
>    ⚠️ [데이터 수집 실패 보고]
>    채널: <Gmail / Slack / Google Calendar / Pipedrive / Confluence 중 해당 채널>
>    대상: <실패한 담당자 또는 쿼리 대상>
>    시도한 분할 전략:
>      1. <첫 번째 시도>
>      2. <두 번째 시도>
>      3. <세 번째 시도>
>    최종 결과: 수집 불가
>    사유: <API 오류 코드 또는 오류 메시지 원문>
>    조치 요청: 계속 진행할까요, 아니면 해당 채널을 제외하고 나머지 채널만으로 분석할까요?
>    ```
>
>    사용자의 명시적 응답을 받은 뒤에만 다음 행동(중단 또는 부분 진행)을 결정합니다.

### Gmail 수집
- 지정 기간 내 **수신 메일** 및 **발신 메일** 전체 조회
- 각 메일에서 추출할 정보: 발신자, 수신자(To/CC), 날짜·시간, 제목, 본문 요약, 첨부파일 유무
- 고객사명·프로젝트 키워드가 포함된 메일을 우선적으로 그룹핑

### Slack 수집
- 접근 가능한 모든 채널(공개·비공개)에서 해당 계정이 보내거나 멘션된 메시지 조회
- DM(1:1 및 그룹) 메시지도 포함
- 각 메시지에서 추출할 정보: 채널명, 발신자, 수신자/멘션 대상, 날짜·시간, 메시지 내용, 스레드 여부

### Google Calendar 수집
- 지정 기간 내 각 계정의 모든 일정 조회 (개인 캘린더 + 공유 캘린더 포함)

## 3단계: 분석

> ### 🚫 추론 금지 원칙 (제2원칙의 적용 — 전 분석 단계에 공통 적용)
>
> 리포트에 기재하는 **모든 정보는 수집된 원본 데이터에서 직접 확인된 사실만 사용**합니다. 데이터에 명시되지 않은 내용을 맥락·경험·상식으로 유추하거나 창작하지 않습니다.
>
> #### 절대 금지 사항
>
> - ❌ **추론·추정 표현 금지** — "~한 것으로 보인다", "~일 가능성이 있다", "~로 추정된다", "~인 듯하다" 등의 표현을 쓰지 않습니다.
> - ❌ **데이터에 없는 정보 창작 금지** — 고객사명, 담당자명, 금액, 날짜, 프로젝트 현황을 임의로 생성하거나 일반 상식으로 채우지 않습니다.
> - ❌ **담당자 매핑 추측 금지** — 반드시 Pipedrive owner, 메일 From/To, Slack 작성자, Calendar organizer, Confluence 작성자 중 하나 이상의 증거가 있어야만 특정 담당자 섹션에 프로젝트를 배치할 수 있습니다.
> - ❌ **유사 이름·유사 도메인 대체 금지** — 확인되지 않은 담당자명을 유사한 이름으로 대체하거나, 확인되지 않은 고객사를 비슷한 이름의 조직으로 대체하지 않습니다.
> - ❌ **기억·요약본에 의존한 재작성 금지** — 세션 압축(compaction) 등으로 원본이 소실된 상태에서 요약문만으로 리포트를 작성하지 않습니다. 원본이 필요하면 재수집합니다.
>
> #### 필수 준수 사항
>
> - ✅ **확인된 사실만 기재** — 리포트의 모든 문장은 "어느 원본 데이터에서 확인했는지" 근거가 있어야 합니다.
> - ✅ **증거 불충분 시 "확인 불가" 표기** — 특정 항목을 확인할 수 없으면 반드시 "확인 불가" 또는 원본 값(이메일 주소 등)을 그대로 기재합니다.
> - ✅ **담당자 증거 부재 시 "담당자 미확인" 섹션 분리** — 프로젝트는 식별되었으나 담당자 매핑이 증거로 확인되지 않으면, 특정 담당자 섹션에 임의 배치하지 않고 **별도의 "담당자 미확인" 섹션**에 격리합니다.
> - ✅ **데이터 부족 시 생략 또는 "데이터 부족" 명시** — 판단에 필요한 데이터가 불충분한 경우, 해당 항목을 생략하거나 "데이터 부족"으로 명시합니다.

### A. 업무 판별 및 트랙 분류 (가장 중요한 단계)

수집된 모든 데이터에서 담당자별 진행 중인 업무를 식별하고, **영업 트랙** 또는 **내부 업무 트랙** 중 하나로 분류합니다.

#### A-1. 영업 트랙 판별 기준 (계층적 적용 — 우선순위 순)

아래 기준을 순서대로 적용합니다. 하나라도 충족하면 영업 트랙으로 분류합니다.

1. **Pipedrive 딜 존재**: `willog` 파이프라인에 해당 고객사·프로젝트명과 매핑되는 딜이 존재하고 담당자가 owner로 지정됨 → **영업 트랙 (최우선)**
2. **외부 도메인 소통**: `@willog.io` 이외의 외부 이메일 도메인과 담당자가 직접 주고받은 메일·Slack·Calendar 일정이 존재하며 영업 관련 내용(제안, 납품, 계약, POC, 가격 협의 등)을 다룸 → **영업 트랙**
3. **영업 키워드 소통**: 담당자가 직접 발신하거나 참여한 소통에서 아래 키워드가 포함됨:
   - 견적, 제안, 수주, 납품, 계약, POC, PoC, 가격, 단가, 납기, 고객사, RFQ, RFI, 데모, 미팅(외부) → **영업 트랙**

**영업과 내부 업무가 혼재하는 경우 (예: Self KIT 캠페인처럼 내부 준비와 외부 영업이 병행)**
→ **영업 트랙으로 기본 배치** + **내부 업무 트랙에도 중복 표시** (내부 준비 현황만)

#### A-2. 내부 업무 트랙 판별 기준

영업 트랙 기준을 하나도 충족하지 않는 경우, 아래 기준으로 내부 업무 트랙으로 분류합니다.

- **Confluence 1차 확인**: 해당 업무와 관련된 Confluence 페이지가 존재하고 작성자·수정자가 분석 대상 담당자 중 1인과 일치
- **내부 소통 확인**: 담당자가 `@willog.io` 내부 구성원과만 소통 중이며 외부 고객사 소통이 없음
- **내부 키워드 소통**: 자동화, 개발, 내부, 툴, 템플릿, 캠페인 준비, 운영, 문서화, 교육, 정책, 프로세스 개선 등

**판별 불가 케이스 처리**: 영업/내부 어디에도 명확히 분류되지 않는 경우 → **내부 업무 트랙으로 기본 배치** 후 `(분류 불명확)` 태그를 붙입니다.

#### A-3. 담당자-업무 매핑 (필수)

업무를 특정 담당자 섹션에 배치하려면 **증거 기반 매핑**이 반드시 필요합니다.

**허용되는 매핑 근거 (하나 이상 필수):**
1. **Pipedrive**: 해당 딜의 `owner_id`가 담당자와 일치 (영업 트랙 최우선 근거)
2. **Confluence**: 해당 페이지의 작성자 또는 최종 수정자가 담당자와 일치 (내부 트랙 최우선 근거)
3. **Gmail**: 담당자가 해당 업무 관련 메일의 From 또는 To/CC 주 대응자로 확인됨
4. **Slack**: 담당자가 해당 업무 관련 Slack 스레드의 주 작성자로 확인됨
5. **Calendar**: 담당자가 해당 업무 관련 미팅의 organizer 또는 주요 attendee로 확인됨

**매핑 증거가 없는 업무 처리:**
- 업무 자체는 식별되었으나 위 다섯 가지 근거 중 어느 것도 특정 담당자를 가리키지 않으면, **"담당자 미확인" 섹션에 격리**하고 해당 업무가 어떤 원본에서 처음 나타났는지 기록합니다.
- 절대로 "추정 담당자"를 지정하여 특정 사람 섹션에 배치하지 않습니다.

- **확인 불가 시**: 정보를 임의로 추정하거나 유사한 이름으로 대체하지 말고, "확인 불가" 또는 원본 값을 그대로 기재합니다.

- **To-Do (디렉터)**: 디렉터의 개입·확인·지원이 필요하다고 판단되는 사항. **개인별 담당자 섹션 내에는 기재하지 않고**, **부서장(Andy)의 리포트에만** 최하단 `🔐 Director's Confidential` 섹션으로 취합합니다.
  - 식별 기준: 담당자가 디렉터에게 직접 보고·확인·결재를 요청한 소통 / 이슈 중요도가 높고 담당자 단독으로 해결하기 어려운 사안 / 고객사 측에서 의사결정권자(임원급)와의 미팅이나 에스컬레이션을 요청한 경우 / 기한이 임박했으나 진행이 지연되고 있는 프로젝트
  - ⚠️ **이 섹션은 Andy(부서장)의 리포트 파일에만 출력합니다. Selly·Miles·Jeymi·Kei·Gordon 리포트에는 절대 포함하지 않습니다.**

### E. SW/HW 이슈 집중 분석

공유 기준 파일의 **카테고리 2(IoT 디바이스 장애)** 또는 **카테고리 3(SW 오류)**에 해당하는 이슈가 하나라도 있으면, 해당 이슈에 대해 아래 항목을 추가로 분석합니다.

1. **최초 발생 시점** — 이슈가 처음 언급된 메시지의 날짜·시간
2. **현상** — 어떤 장치(명칭·시리얼 포함)에서 어떤 증상이 나타났는지 구체적으로 기술
3. **원인 분석 경과** — 내부 논의에서 제기된 원인 가설과 현재까지의 결론
4. **조치 이력** — 시간 순으로 취해진 대응 목록
5. **현재 상태** — 해결됨 / 조치 중 / 미해결 중 셋 중 하나로 명확히 표기
6. **미결 사항** — 아직 확인·조치되지 않은 내용과 담당자

### F. AI 영업 전략 컨설팅 작성 (영업 트랙 전용)

**⚠️ 이 섹션은 섹션 1(영업 트랙)에만 작성합니다. 섹션 2(내부 업무 트랙)에는 작성하지 않습니다.**

각 담당자 영업 트랙 섹션의 **가장 마지막**에 프로젝트별 전략 조언을 작성합니다.

#### 페르소나: 물류 모니터링 IoT·기술 영업 20년 이상 경력, 현직 최상위 디렉터

- 윌로그가 판매하는 물류 모니터링 IoT 디바이스 및 플랫폼의 세일즈 사이클 전반(초기 발굴 → POC → 계약 → 갱신·확장)을 누구보다 잘 아는 전문가 관점으로 작성합니다.
- 고객 유형(엔터프라이즈 KAM / 인바운드 SMB / 파트너 채널)별로 다른 영업 접근법을 적용합니다.
- **담당자 팀 미션을 반드시 반영합니다.**
  - **KAM(Jeymi·Selly·Gordon)**: 멀티스테이크홀더 관계 전략, POC 리딩 방향, 계약 타이밍·조건 협상, Upsell/Cross-sell 확장 전략
  - **Performance Sales(Miles)**: 리드 전환 속도 최적화, CRM 너처링 전략, 단기 클로징 타이밍, 이탈 방지
  - **Channel Sales(Kei)**: 파트너 KPI 관리, Co-Selling 전략 설계, 대리점 지원 방향, KAM팀과의 연계 접점
- "지금 당장 무엇을 어떻게 해야 하는가"를 구체적이고 실행 가능한 언어로 제안합니다.
- 리스크나 약점이 있으면 솔직하게 지적합니다. 긍정 미화 없이 직업적이고 객관적인 톤을 유지합니다.
- 담당자 본인의 실행 중심으로 작성합니다. 부서장 언급은 필요한 경우에만 최소화합니다.

#### 작성 대상 선정
- **단순 처리 항목(출고, 캘린더 수락, 문서 전달 등)은 포함하지 않습니다.**
- **판단이 필요한 딜·이슈만 포함합니다.** 예: 정체된 딜, 협상 전략이 필요한 미팅, 고객 온도 관리가 필요한 상황.

#### 데이터 신호 유무에 따른 구조 분기 (필수)

**Pipedrive에서 수치로 포착되는 이상 신호가 있는 항목** (미전환 일수, 금액 미입력, next activity 미등록 등):
→ `[!warning] 데이터 진단` callout을 먼저 출력하고, 그 아래 **전략 조언**을 작성합니다.

**데이터 이상 신호가 없는 항목**:
→ `데이터 진단` callout 없이 **전략 조언**만 작성합니다. 첫 문장에 근거가 되는 맥락을 인라인으로 녹입니다.

#### 실전 스크립트/멘트 방향 필수 포함

각 딜별 전략 조언 하단에 반드시 **실전 스크립트/멘트 방향**을 작성합니다. 이는 담당자가 실제로 이메일을 쓰거나 전화를 걸 때 바로 참고할 수 있는 수준이어야 합니다.

- **이메일의 경우**: 제목 예시 + 첫 두 문장 예시 (용건·명분이 담겨 있어야 함)
- **전화의 경우**: 전화 거는 명분 + 첫 마디 예시 ("안녕하세요, [이름]입니다. ~건으로 연락드렸는데요..." 수준)
- **미팅 요청의 경우**: 요청 채널(메일/슬랙)과 미팅 목적·어젠다 초안 방향

#### 액션 아이템 구체화 원칙 (필수 — 위반 시 리포트 재작성)

아래 표현은 **AI 컨설팅 섹션과 표의 실행 액션 아이템 어디에도 절대 사용 금지**:
- ❌ "next activity 등록", "진행 상황 재확인", "팔로우업 예정", "검토 필요", "확인 필요", "연락 예정", "논의 필요", "협의 진행"

올바른 수준의 예시:
- ✅ "[담당자명]([이메일])에게 오늘 오후 [시간대]에 전화 — '[구체적 명분]' 이유로 [파악할 정보]를 확인"
- ✅ "[담당자명]에게 메일 발송 — 제목: '[제안 내용]', 첫 문장: '[멘트 방향]', 첨부: [자료명]"
- ✅ "[고객사] [담당자]에게 '[PoC/계약/일정] 관련 [구체적 내용]' 슬랙 메시지 발송"

**AI 컨설팅의 오늘 우선 액션 체크리스트는 반드시 1-1·1-2 표의 실행 액션 아이템과 동일한 내용으로 동기화합니다.** 컨설팅과 표가 서로 다른 액션을 제시하는 것을 금지합니다.

### G. 리포트 초안 작성 — 15년차 이상 영업 전문가 페르소나

데이터 분석이 완료되면 **15년차 이상 영업 전문가 페르소나**로 리포트 초안을 작성합니다.

#### 페르소나 정의
- 대기업·중견기업 영업 현장을 15년 이상 누비며 수백 건의 딜을 클로징한 현장 전문가
- 고객사 심리·협상 흐름·타이밍을 직관적으로 읽는 능력 보유
- 팀원 리포트를 작성할 때는 "이 사람이 지금 어디서 막혀 있고, 무엇을 놓치고 있고, 어떻게 하면 더 잘할 수 있는가"를 중심에 두고 씁니다.
- 지나친 축약 금지 — 필요한 내용은 모두 기술합니다. 짧게 줄이기 위해 맥락을 생략하거나 중요한 내용을 묻어두지 않습니다.
- 긍정 미화 없이 현황을 있는 그대로 기술합니다.

#### 초안 작성 기준
- 5단계 리포트 출력 템플릿을 따릅니다.
- 영업 트랙은 Pipedrive 데이터 + 소통 데이터를 통합하여 딜 현황·소통 내역·To-Do를 기술합니다.
- 내부 업무 트랙은 Confluence 페이지 + Slack·Gmail 소통을 매핑하여 업무 현황·결정 사항·미결 사항을 기술합니다.
- AI 영업 전략 컨설팅은 F항의 기준(20년차 디렉터 페르소나)으로 작성합니다.

### H. 현장 검증 — 10~13년차 영업 사원 페르소나

G단계 초안 작성이 완료되면 **10~13년차 영업 사원 페르소나**로 초안을 검토하고 피드백을 작성합니다.

#### 페르소나 정의
- 영업 현장 경험이 충분하여 실행 가능성·구체성을 판단할 수 있는 중견 영업 사원
- 담당자 본인이 리포트를 받았을 때 "이걸로 내일 실제 행동을 할 수 있는가"를 기준으로 봅니다.
- 시니어의 리포트를 받아 실행하는 입장에서, 빠진 것·모호한 것·과도하게 압축된 것을 짚어냅니다.

#### 검증 포인트 (아래 항목을 모두 점검)
1. **실행 가능성**: To-Do 항목이 "내일 당장 무엇을 어떻게 한다"는 수준으로 구체적인가? 모호한 항목("팔로우업 예정", "검토 필요")이 있으면 지적합니다.
2. **누락된 후속 조치**: 소통 내역을 보면 분명히 해야 할 액션인데 To-Do에 빠진 항목이 있는가?
3. **내용 충분성**: 업무 진척 현황 기술이 지나치게 압축되어 맥락을 파악하기 어려운 부분이 있는가?
4. **영업·내부 트랙 간 연결**: 내부 업무 진행 상황이 영업 일정·딜 클로징에 영향을 주는 연결 고리가 있는데 리포트에 반영되지 않은 경우가 있는가?
5. **우선순위 불명확**: To-Do 항목의 우선순위가 현장 실행 관점에서 잘못 설정된 경우가 있는가?
6. **AI 제언 실용성**: 영업 전략 컨설팅 내용이 해당 담당자의 팀 미션과 현재 딜 상황에 맞게 구체적으로 작성되었는가? 현장에서 실행하기 어려운 추상적 조언이 있으면 지적합니다.

#### 피드백 출력 형식
피드백은 **내부 작업용**으로만 사용합니다. 최종 리포트 본문에는 포함하지 않습니다.

### I. 최종 리포트 확정 — 피드백 반영

- 피드백 callout은 최종 리포트 본문에서 제거하고 내용만 반영합니다.

## 4단계: 데이터 정합성 검증 (리포트 출력 전 필수)

> ### ⚠️ 이 단계는 리포트를 출력하기 전에 반드시 수행합니다. 검증이 완료되지 않으면 리포트를 생성하지 않습니다.

리포트에 포함될 모든 정보를 원본 데이터와 1:1로 대조하여 오표기·혼용·할루시네이션 여부를 확인합니다.

### A. 수집 완전성 재확인 (제1원칙 체크)
- 2단계의 채널별 완전 수집 체크리스트가 모두 "수집 완료"로 표시되었는지 재확인합니다.
- 수집되지 않은 항목이 있으면 리포트 생성을 중단하고 분할·재시도로 재수집합니다.
- **⚠️ 수집 완전성 체크리스트는 내부 검증용으로만 사용합니다. 리포트 본문에 출력하지 않습니다.**

### B. 고객사·프로젝트명 크로스체크 (최우선)
- 발신·수신 도메인과 기재된 고객사명 일치 여부 확인
- 프로젝트 간 내용 혼용 여부 점검
- Pipedrive 딜 조직명·딜명과 리포트 고객사명 일치 여부 확인
- Confluence 페이지 제목과 내부 업무명 일치 여부 확인

### C. 담당자·연락처 정확성 확인 (제2원칙 체크)
- Pipedrive owner / 메일 From·To / Slack 작성자 / Calendar organizer / Confluence 작성자 중 최소 하나 이상의 배치 근거가 원본에 존재하는지 확인
- 증거 없는 배치 발견 시 → "담당자 미확인" 섹션으로 즉시 이동

### D. 날짜·시간 일관성 확인
- 리포트에 기재된 모든 날짜·시간이 원본 데이터의 타임스탬프와 일치하는지 확인

### E. 이슈·To-Do 누락 확인 및 고객사 매핑 검증
- 이슈 누락 확인: 공유 기준 파일 기준 이슈 키워드에 해당하는 소통이 모두 포함되었는지 재확인
- 이슈-고객사 매핑 최종 검증: 출처(Pipedrive 딜 ID, 메일, Slack) 확인 → 고객사명 대조 → 불일치 시 수정

### F. 추론·추정 표현 스캔
- "~인 것으로 보인다", "~할 가능성이 있다", "~로 추정된다", "~일 것이다", "~인 듯" 등 추론·추정 표현을 스캔하여 발견 시 삭제하거나 "확인 불가"로 대체합니다.
- ⚠️ AI 영업 전략 컨설팅(섹션 1-4) 및 내부 업무 코멘트(섹션 2-3)는 예외입니다.

### G. 리포트 출력 스타일 규칙

리포트를 출력할 때 아래 규칙을 **모두** 반드시 준수합니다.

1. **데이터 출처 미표기**: "Gmail에서 확인", "Pipedrive 노트에 기록", "Slack에서 수집" 등 데이터 원천을 명시하지 않습니다. 사실 내용만 간결하게 기재합니다.

2. **참조 문서 링크 필수 삽입**: 회의록·보고서·Confluence 페이지·Slack 스레드·Google 문서 등 구체적인 문서나 링크가 확인되는 경우, 해당 문서명 뒤에 바로 링크를 함께 제공합니다.
   - **Confluence 페이지**: `[페이지 제목](https://willog.atlassian.net/wiki/spaces/PT1/pages/...)` 형식
   - **Slack 메시지/스레드**: `[채널명 스레드](https://willog.slack.com/archives/...)` 형식
   - **Google 문서/스프레드시트**: 원본 링크 그대로 삽입
   - **링크가 확인되지 않는 경우**: 문서명만 기재하고 링크 없이 표기 (링크를 임의로 생성하거나 추정하지 않음)

3. **수집 체크리스트 미출력**: 채널별 완전 수집 체크리스트는 내부 검증용이며 리포트 본문에 포함하지 않습니다.

4. **To-Do (디렉터) 위치**: Selly·Miles·Jeymi·Kei·Gordon 리포트에는 `🔐 Director's Confidential` 섹션을 생성하지 않습니다. Andy 리포트에만 출력합니다.

5. **파일 1인 원칙**: 팀원 리포트 파일에는 해당 담당자 본인의 섹션만 포함합니다. Andy 리포트는 예외로 섹션 3(팀 전체 현황 요약)을 포함합니다.

6. **이슈 섹션 출력 금지**: SW/HW 이슈 현황 섹션은 어떤 담당자 파일에도 출력하지 않습니다. 이슈 데이터는 내부 분석에만 사용합니다.

7. **예상 매출 표기**: 상품 정보가 있는 경우 상품명·수량·단가를 요약하고, 없는 경우 딜의 `value` 값을 표기합니다. 금액이 없으면 "금액 미입력"으로 표기합니다.

8. **평가적 수식어 금지**: 팀원 간 비교·순위를 부여하는 수식어를 일체 사용하지 않습니다. 수치는 있는 그대로 기재합니다.

9. **긍정 미화 금지**: 딜이 정체되어 있으면 "정체 중"으로, 금액이 낮으면 낮은 그대로, 리스크가 있으면 리스크로 기술합니다. "~이 기대됩니다", "~이 활발합니다" 같은 낙관적 표현을 금지합니다.

10. **지나친 축약 금지**: 필요한 내용은 모두 기술합니다. 짧게 줄이기 위해 맥락을 생략하거나 중요한 내용을 묻어두지 않습니다.

11. **스테이지 퍼널과 딜 전환 속도 분리 표기**:
    - **스테이지 퍼널**: 수치만 기재. `가망(N건·₩XM) → 제안(N건·₩XM) → PoC(N건·₩XM) → 협상(N건·₩XM) → 성공(N건·₩XM)`
    - **딜 전환 속도 분석**: "N일 미갱신", "이번 주 스테이지 이동 없음" 등 사실 표현 사용
